# 设计：降低 CLS 与提升运行时性能（修正版 v2）

## 设计目标

在不改变视觉设计和功能行为的前提下：
1. 消除 `.site-frame` padding transition 导致的 CLS
2. 将 `scroll` + `getBoundingClientRect` 替换为 `IntersectionObserver`，提升滚动帧率
3. 缩短首屏关键请求链
4. 降低动画 GPU 开销

---

## 设计一：消除 CLS — 移除 `.site-frame` 的 padding transition

### 当前方案

```css
/* global.css L119 */
.site-frame {
  padding-inline-start: calc(1.25rem + var(--sidebar-offset));
  transition: padding-inline-start 320ms ease;  /* 导致 CLS */
}
```

```tsx
// DynamicSidebar.tsx L529-533
useEffect(() => {
  if (window.innerWidth >= 1024) {
    setCollapsed(false);  // 触发 syncDockLayout → 修改 --sidebar-offset
  }
}, []);
```

### 问题

当 `DynamicSidebar` 在组件挂载时展开，`--sidebar-offset` 从 `0px` 变为 `~332px`，`.site-frame` 的 `padding-inline-start` 产生 320ms 的 transition 动画，导致 9 次连续布局偏移。

### 新方案

**方案 A（推荐）：服务器端预设 `--sidebar-offset`**

在 `SiteLayout.astro` 的 `<script is:inline>` 中，根据视口宽度预设 `--sidebar-offset`：

```html
<script is:inline>
  try {
    // ... 现有 theme/locale 逻辑 ...
    
    // 新增：预设 sidebar offset，避免客户端动画
    const sidebarWidth = 304; // 19rem ≈ 304px
    const sidebarOffset = window.innerWidth >= 1180 ? `${sidebarWidth + 28}px` : '0px';
    document.documentElement.style.setProperty('--sidebar-offset', sidebarOffset);
  } catch {
    // ...
  }
</script>
```

同时从 `.site-frame` 中移除 `transition: padding-inline-start`：

```css
.site-frame {
  padding-inline-start: calc(1.25rem + var(--sidebar-offset));
  /* 移除 transition: padding-inline-start 320ms ease */
}
```

**优势**：
- 页面渲染前即确定正确的 padding，无动画过程
- 完全消除由 padding transition 导致的 CLS
- 保留 sidebar 自身的 `transform` 展开动画（视觉反馈仍在）

**方案 B（备选）：延迟 sidebar 展开**

在 `DynamicSidebar` 中延迟设置 `collapsed: false`，等待页面完全加载：

```tsx
useEffect(() => {
  if (window.innerWidth >= 1024) {
    // 使用 requestAnimationFrame 或 setTimeout 延迟展开
    const timer = requestAnimationFrame(() => {
      setCollapsed(false);
    });
    return () => cancelAnimationFrame(timer);
  }
}, []);
```

**劣势**：sidebar 展开有延迟，用户体验稍差。

## 设计二：DynamicSidebar TOC 追踪重构（运行时性能）

### 当前方案

- 监听 `window` 的 `scroll` 事件
- 在事件处理中调用 `heading.element.getBoundingClientRect().top`
- 使用 `requestAnimationFrame` 调度计算

### 问题

`getBoundingClientRect()` 强制浏览器同步计算布局，阻塞主线程，影响滚动帧率。

**注意**：这是运行时性能问题，不是 CLS 问题。`getBoundingClientRect()` 是只读 API，不会修改元素几何属性。

### 新方案

采用 `IntersectionObserver` 替代滚动监听。

#### 架构图

```
┌─────────────────────────────────────────────────────────────┐
│  旧架构：scroll 事件驱动（影响帧率）                            │
│                                                             │
│  window.scroll ──► handleWindowScroll ──► rAF              │
│                                              │              │
│                                              ▼              │
│                                    getBoundingClientRect()  │
│                                    （强制同步布局，阻塞主线程）   │
│                                              │              │
│                                              ▼              │
│                                    setActiveId (state 更新)  │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│  新架构：IntersectionObserver 驱动（异步，不阻塞）              │
│                                                             │
│  IntersectionObserver(rootMargin: '-14% 0px -68% 0px')     │
│           │                                                 │
│           ▼                                                 │
│  entries ──► syncActiveHeading ──► setActiveId             │
│                                                             │
│  优势：浏览器异步计算，不阻塞主线程，无强制同步布局              │
└─────────────────────────────────────────────────────────────┘
```

#### 关键代码结构

```tsx
// 新方案核心逻辑
useEffect(() => {
  if (flatItems.length === 0) return;

  const elements = flatItems
    .map((item) => document.getElementById(item.slug))
    .filter((el): el is HTMLElement => Boolean(el));

  const observer = new IntersectionObserver(
    (entries) => {
      // 如果处于点击导航锁定状态，忽略 observer 更新
      if (pendingScrollTargetRef.current) return;

      const visible = entries
        .filter((e) => e.isIntersecting)
        .sort((a, b) => a.boundingClientRect.top - b.boundingClientRect.top);

      if (visible[0]) {
        setActiveId(visible[0].target.id);
        return;
      }

      const crossed = entries
        .filter((e) => e.boundingClientRect.top < ACTIVE_LINE_PX)
        .sort((a, b) => b.boundingClientRect.top - a.boundingClientRect.top);

      if (crossed[0]) {
        setActiveId(crossed[0].target.id);
      }
    },
    {
      rootMargin: '-14% 0px -68% 0px',
      threshold: [0, 0.2, 0.4, 1],
    }
  );

  elements.forEach((el) => observer.observe(el));
  return () => observer.disconnect();
}, [flatItems]);
```

#### 点击导航锁定机制（保留）

```tsx
// 点击 TOC 链接后，设置锁定状态
const handleTocNavigation = (id: string, event: ReactMouseEvent<HTMLAnchorElement>) => {
  event.preventDefault();
  
  // 设置 pending 状态，阻止 IntersectionObserver 更新覆盖
  pendingScrollTargetRef.current = id;
  setActiveId(id);
  
  // 滚动到目标
  target.scrollIntoView({ behavior: prefersReducedMotion ? 'auto' : 'smooth' });
  
  // 1.6 秒后自动释放锁定
  maxPendingTimeoutRef.current = window.setTimeout(() => {
    pendingScrollTargetRef.current = null;
  }, MAX_PENDING_MS);
};
```

**注意**：`pendingScrollTargetRef` 是锁定机制的核心状态，**不可移除**。初版设计文档错误地建议移除它，这会导致点击导航后 TOC 高亮被滚动事件覆盖。

## 设计三：Islands 加载策略调整

### 当前策略

| 组件 | 指令 | 加载时机 |
|------|------|----------|
| DynamicSidebar | `client:load` | DOM Ready 立即加载 |
| CompactTimeline | `client:load` | DOM Ready 立即加载 |
| GalleryFilter | `client:visible` | 进入视口时加载 |

### 问题

`client:load` 在页面加载时立即水合 React 运行时，拉长了关键请求链。

### 新策略

| 组件 | 新指令 | 加载时机 | 理由 |
|------|--------|----------|------|
| DynamicSidebar | `client:idle` | 浏览器空闲时 | 首屏可见但非交互关键 |
| CompactTimeline | `client:visible` | 滚动到时间线区域 | 通常不在首屏 |
| GalleryFilter | `client:visible` | 保持现状 | 已在视口内延迟加载 |

**注意**：Astro 的 `client:idle` 具体实现方式未在官方文档中明确说明，可能使用 `requestIdleCallback` 或自定义的 `setTimeout` 降级方案。

## 设计四：动画性能优化

### 当前实现

```css
.ambient-ribbon {
  background: var(--ambient-ribbon);
  animation: ambient-breathe 11s ease-in-out infinite;
  filter: blur(18px);  /* 无法 GPU 加速 */
}
```

### 问题

`filter: blur()` 无法被 GPU 加速，动画期间每帧都需要 CPU 计算模糊效果。

### 新实现

```css
.ambient-ribbon {
  background: var(--ambient-ribbon);
  animation: ambient-breathe 11s ease-in-out infinite;
  /* 移除 filter: blur(18px) */
  will-change: transform, opacity;  /* 提示浏览器优化 */
}
```

替代方案：使用预模糊的 SVG 滤镜或背景图替代实时 CSS `filter`。

## 回滚策略

若方案 A（服务器端预设 `--sidebar-offset`）导致响应式布局问题：

1. 恢复 `.site-frame` 的 `transition`，但将其改为仅在大屏下生效：
   ```css
   @media (min-width: 1180px) {
     .site-frame {
       transition: padding-inline-start 320ms ease;
     }
   }
   ```
2. 在 `DynamicSidebar` 中使用 `useLayoutEffect` 而非 `useEffect` 设置初始状态，确保在绘制前完成布局计算

若 `IntersectionObserver` 方案导致 TOC 高亮不准确：

1. 恢复 `scroll` 事件监听
2. 引入节流函数（100ms），减少 `getBoundingClientRect()` 调用频率
3. 缓存 `getBoundingClientRect()` 结果，避免在单帧内重复计算
