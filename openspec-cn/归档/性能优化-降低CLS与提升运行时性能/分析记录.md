## 关键修正说明

### 初版错误 #1（已修正）

**错误陈述**：`getBoundingClientRect()` 在滚动事件中触发，导致 CLS。

**事实**：`getBoundingClientRect()` 是只读 API，不会修改元素几何属性，因此**不会直接导致 CLS**。它导致的是**强制同步布局（Forced Reflow）**，影响的是**滚动帧率/运行时性能**，而非累积布局偏移。

**CLS 真实成因**：通过重新录制 Layout Shift 区域并分析 DOM，发现 CLS 的主要来源是：

1. **`.site-frame` 的 `padding-inline-start` transition**（[global.css#L119](file:///f:/codepository/2026311%20132635/Portfolio%20Studio/src/styles/global.css#L119)）：
   - `transition: padding-inline-start 320ms ease`
   - 当 `DynamicSidebar` 在 `useEffect` 中根据视口宽度设置 `collapsed` 状态时（[DynamicSidebar.tsx#L529-533](file:///f:/codepository/2026311%20132635/Portfolio%20Studio/src/components/DynamicSidebar.tsx#L529-533)），会触发 `syncDockLayout()`，进而修改 CSS 变量 `--sidebar-offset`
   - 这导致 `.site-frame` 的 `padding-inline-start` 从 `0px` 动画过渡到 `~380px`，产生大量小幅度布局偏移（9 次偏移，累积约 0.05）

2. **字体加载导致的文本重排**：
   - `IBM Plex Sans Variable` 和 `IBM Plex Mono` 字体在加载前后，文本尺寸发生变化
   - 虽然字体已使用 `font-display: swap`，但字重变化仍会导致轻微布局偏移

3. **GalleryFilter 初始动画**：
   - 使用 `AnimatePresence` + `motion.a` 的入场动画（`opacity:0; transform:translateY(18px) scale(0.985)`）
   - 元素从不可见到可见的过程中，尺寸计算变化导致布局偏移

### 初版错误 #2（已修正）

**错误陈述**：`TocNav.tsx` 使用了 `IntersectionObserver` 且 "无强制同步布局问题"，可作为正确实现的对比参考。

**事实**：`TocNav.tsx` 确实使用了 `IntersectionObserver`，但它**只在项目详情页中使用**。而 CLS 数据（0.05）是针对**首页**录制的（LCP 元素是 H1 标题文本，位于首页 HeroBanner）。将 `TocNav.tsx` 作为解决首页 CLS 问题的参照，逻辑关联性不足。

### 初版错误 #3（已修正）

**错误陈述**：DOM 规模膨胀归因于 "SVG 封面图片内嵌复杂节点"。

**事实**：代码库中项目封面使用的是 `<img>` 标签加载外部图片文件（如 `cover.svg`），**不是内嵌 SVG DOM**。DOM 中的 SVG 元素主要是 `lucide-react` 图标（24x24，少量 path 节点）。Performance Trace 中提到的 "circle 元素"、"filter#g 元素" 可能来自某个图标的 SVG 结构，但与项目封面无关。

### 初版错误 #5（已修正）

**错误陈述**：设计文档中 `IntersectionObserver` 方案缺少 `pendingScrollTargetRef` 的处理，且规格说明和设计文档在保留/移除 `pendingScrollTargetRef` 上存在矛盾。

**事实**：经代码审查确认，`pendingScrollTargetRef` 是点击 TOC 导航后的**锁定机制核心状态**，用于防止滚动期间 observer 更新覆盖用户点击的目标。如果移除 `pendingScrollTargetRef`，`maxPendingTimeoutRef` 的锁定机制就无法工作。设计文档的代码示例确实缺失了这一关键逻辑。

### 初版错误 #6（已修正）

**错误陈述**：`client:idle` 使用 `requestIdleCallback` 实现，且 "Astro 已内置 polyfill"。

**事实**：Astro 官方文档未明确说明 `client:idle` 的内部实现方式。`requestIdleCallback` 在 Safari 中支持较晚，Astro 更可能使用自定义的 `setTimeout` 降级方案。关于 "内置 polyfill" 的说法缺乏官方文档支持。

### 初版错误 #7（已修正）

**错误陈述**：`GalleryFilter` 缺少 `aspect-square`，需要添加。

**事实**：`GalleryFilter.tsx#L108` 已经存在 `<div className="relative aspect-square overflow-hidden">`。任务清单中的要求与代码现状不符。

### 初版错误 #9（已修正）

**错误陈述**：引用了 `/.ai_context/report.html` 和 `/.ai_context/performance-trace.json.json.gz` 作为相关分析资料。

**事实**：这些文件是性能分析过程中临时生成的，不应作为提案的正式引用资料。

## 代码审查发现（修正后）

### DynamicSidebar.tsx

**CLS 相关**：
- `useEffect`（L529-533）在挂载时根据 `window.innerWidth >= 1024` 设置 `collapsed: false`
- 这触发 `syncDockLayout()` → 设置 `--sidebar-offset` → `.site-frame` 的 `padding-inline-start` 产生 320ms 的 transition 动画
- **这是首页 CLS 的主要来源**

**运行时性能相关**：
- `computeActiveHeading` 中使用 `getBoundingClientRect()`（L367-368）
- 该函数通过 `requestAnimationFrame` 调度（L427-432）
- 滚动事件监听器使用 `{ passive: true }`（L607）
- **这影响的是滚动帧率，不是 CLS**

### global.css

- `.site-frame` 的 `padding-inline-start` 有 `320ms ease` 的 transition（L119）
- `.ambient-ribbon` 使用 `filter: blur(18px)` + `animation`（L152-154）
- `body` 使用 `transition: background-color 220ms ease, color 220ms ease`（L68）

### index.astro

- `DynamicSidebar client:load` — 立即加载，触发 sidebar 展开动画
- `CompactTimeline client:load` — 立即加载
- `GalleryFilter client:visible` — 视口内加载（画廊区域通常在首屏或首屏下方）

## 结论（修正后）

主要性能瓶颈集中在：

1. **CLS 来源**：`.site-frame` 的 `padding-inline-start` transition（由 DynamicSidebar 初始展开触发）
2. **运行时性能**：`getBoundingClientRect()` 导致的强制同步布局（影响滚动帧率）
3. **首屏加载**：`client:load` 过早加载 React 运行时
4. **动画 GPU 开销**：`filter: blur()` 无法 GPU 加速

优化后预期：

- CLS 从 ~0.05 降至 < 0.01（移除 padding transition 或延迟 sidebar 展开）
- 滚动帧率稳定在 60fps（使用 IntersectionObserver 替代 scroll + getBoundingClientRect）
- 关键请求链缩短 30%+
